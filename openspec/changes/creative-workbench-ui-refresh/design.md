## Context

当前前端在视觉上同时依赖三类来源：

- `frontend/src/index.css` 中的 Tailwind v4 `@theme` token
- `frontend/src/css/app.css` 与 `frontend/src/css/styles.css` 中的遗留全局样式
- 大量组件内部直接写死的 `bg-gray-*`、`text-gray-*`、`border-gray-*`、`indigo-*` 类名

这种混合方式导致两个直接后果：

1. 同一层级的页面和组件使用不同的背景、边框和强调色规则，整体观感像多个阶段的暗色后台叠加在一起
2. 颜色和信息层级耦合在组件内部，后续想统一风格时只能逐组件清点，缺乏稳定的设计基线

本次变更不引入新的业务流程，也不改变路由或后端 API。它是一次前端跨页面、跨模块的设计系统收敛，影响项目大厅、工作区壳层、系统配置页以及这些页面共享的表面与状态语义。

## Goals / Non-Goals

**Goals:**

- 建立一套以石板蓝黑、冷白、雾蓝灰为主的统一视觉 token，并明确高亮色与状态色职责
- 让项目大厅、全局头部、工作区三栏和系统配置页共享同一套表面层级与文字层级，而不是各自写死深灰与 indigo
- 将 TokyoNight Storm 转化为产品界面语言，而不是直接照搬编辑器配色
- 提高高密度页面的可读性，使摘要、操作、警示、进阶区域更易扫描
- 采用渐进替换策略，优先刷新壳层与关键页面，再向局部组件扩散

**Non-Goals:**

- 不调整后端接口、状态字段或业务行为
- 不在本次变更中重做所有业务组件的结构，只聚焦壳层、入口页和高频配置页
- 不追求品牌重塑或 logo 改版，本次只重构产品界面表达
- 不实现独立的明暗主题切换，先把默认深色工作台做完整

## Decisions

### 1. 采用语义 token 层，而不是继续在组件里直接写色值或 Tailwind 灰阶

将新增一套语义化 token，用于表达背景、面板、浮层、边界、主文字、次文字、强调色、成功、警告、危险等角色。

理由：

- TokyoNight Storm 的价值在于层级规则，不在于单个颜色值
- 语义 token 可以同时服务 Tailwind class 与遗留 CSS，避免两套主题并存
- 后续新增页面只需要选用角色，不需要再做一次颜色设计

备选方案：

- 直接在组件里把 `gray-950` 改成蓝黑、把 `indigo-500` 改成亮蓝
- 否决原因：只能得到“换色版旧后台”，无法解决层级混乱与后续漂移问题

### 2. 用“壳层优先”的刷新顺序，而不是一次性细修所有页面

实施顺序定为：

1. 全局主题入口与基础 surface token
2. 项目大厅、全局头部、三栏工作区外壳
3. 系统配置页与其高密度表单区域
4. 再向局部复用组件收敛

理由：

- 用户第一眼感知主要来自壳层，而不是局部控件
- 壳层统一之后，即使局部组件仍未完全重绘，整体质感也会先提升
- 风险更小，能快速验证新主题是否真正符合创作工作台定位

备选方案：

- 先把单个设置表单页做漂亮，再回头调整全局
- 否决原因：页内再精致，也会被旧 header、旧 sidebar、旧项目大厅拖回后台感

### 3. 保留现有信息架构，优先重组视觉节奏与表面层级

本次不推翻项目大厅入口、设置页分区或工作区三栏布局，而是在现有结构上重做：

- 背景与 panel 的明度台阶
- 文本层级与标签层级
- hover / focus / active 的统一反馈
- 卡片、表单、警告、摘要块的表面密度

理由：

- 当前问题主要在表达层，不在 IA 本身
- 结构不大改可以显著降低回归范围
- 更适合与现有测试、状态管理和组件拆分并行演进

备选方案：

- 同时重构页面布局与视觉系统
- 否决原因：变量过多，不利于判断问题究竟来自布局还是主题

### 4. TokyoNight Storm 只作为参考基线，产品内强调色职责重新分配

建议的职责映射为：

- 背景 / 面板：石板蓝黑与蓝灰台阶
- 主文字：冷白
- 次文字 / 注释 / 辅助说明：雾蓝灰
- 主强调色：偏蓝，用于主操作、选中、当前上下文
- 次强调色：偏青，用于 hover、可交互反馈、轻量高亮
- 警告 / 危险：橙、红只用于状态，不混入普通操作

理由：

- 纯把紫色、蓝色、青色都当主按钮色，会再次制造噪音
- ArcReel 是创作工作台，强调色必须节制，不能回到霓虹赛博风过饱和问题

备选方案：

- 继续沿用现有 neon 绿 + indigo 的双主色结构
- 否决原因：现有双主色已经让交互优先级变得模糊，高级感不足

### 5. 将系统配置页视为“工作台内设置中心”，而不是普通后台表单页

设置页将明确区分：

- 页面摘要区
- 全局警示区
- 导航区
- 主要控制区
- 进阶与辅助说明区

并通过新的 panel、边界、文字层级重新组织阅读节奏。

理由：

- 当前问题不在字段缺少，而在整个页面像一堆同层灰卡片连续堆叠
- 这是最容易体现“不是 CRUD 后台”的页面之一，值得作为高密度页面样板

备选方案：

- 仅调整颜色，不改表面分组和密度
- 否决原因：最终只会是“更蓝一点的表单页”

## Risks / Trade-offs

- [风险] 全局 token 收敛会触发大量视觉回归 → **缓解**：先限定在壳层与关键页，避免一次扫全站
- [风险] 遗留 CSS 与 Tailwind class 混用，可能出现局部覆盖异常 → **缓解**：先统一 token 命名，再清理高频冲突入口
- [风险] TokyoNight 风格过度靠近编辑器主题，导致产品感不足 → **缓解**：只借鉴色阶与语义，不照抄语法高亮色分配
- [风险] 强调色收敛后，部分旧页面会显得“没有按钮存在感” → **缓解**：为 primary / secondary / warning / danger 明确交互态规则
- [风险] 设计升级后，旧截图、旧文档与测试快照需要同步 → **缓解**：按页面分阶段更新，避免一次性大范围快照震荡

## Migration Plan

1. 在主题入口新增语义 token，并保持旧 class 暂时可运行
2. 优先替换项目大厅、全局头部、工作区壳层与系统配置页的表面和文字层级
3. 将设置页作为高密度表单样板，验证新的摘要 / 警示 / 控制区表达
4. 根据新 token 回收遗留 `app.css` / `styles.css` 中与新规则冲突的颜色定义
5. 通过视觉验收后，再向复用组件和其余页面扩散

回滚策略：

- 主题 token 保持集中定义，若风格方向需要回退，只需恢复入口 token 与壳层页面样式，不需要回滚业务逻辑

## Open Questions

- 登录页是否一并纳入首批工作台视觉语言，还是保留更克制的简化版入口风格
- 资产侧栏与 Copilot 面板是否需要在同一轮中完成表面统一，还是等壳层稳定后第二轮处理
- 是否需要为项目卡片增加更强的封面氛围区与元信息层级，而不仅是主题换色
